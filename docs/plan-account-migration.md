# アカウント移行計画（会社 → 私物）

## 目的

「暮らしnote」(旧名:わが家ホーム)を日常使いの私物アプリとして長く使うため、会社アカウント（退職で削除され得る）に紐づく資産を私物アカウントへ移し、会社を離れても壊れない状態にする。

## 結論（2026-07-15 本人確認済み）

- **Google（GAS＋スプレッドシート＋家族データ）＝私物アカウント** → 移行不要 ✅（ただし `/u/1/` の多アカウント状態だったので、GASプロジェクトとシートの**所有者が私物か**だけ最終確認推奨）
- **LINE＝個人LINE** → そのままでOK ✅
- **git のコミット作者＝GitHub no-reply**（会社メール非埋め込み）→ 変更不要 ✅
- **GitHub＝会社メールで登録** → **唯一の対応。アカウント/URLは温存し、登録メールを私物に差し替えるだけ**（下記「② GitHub」参照）。移行・URL変更・LINE更新・コード変更はすべて不要。

以降の「移行手順」は、将来フル移行が必要になった場合の参考として残す。今回の実対応は GitHub のメール差し替えのみ。

## 先に確認してほしい3点

1. **Google**: GAS/スプレッドシートは `～@designone.jp`（会社Workspace）で作った？ → その場合は**移行必須**（Workspaceアカウントは退職で削除される）。
2. **GitHub**: `watanabejunsuke-rgb` の登録メール（Settings → Emails）は会社メール？ 私物メールに変更でき、2FAの復旧手段が会社に依存していなければ、**アカウントごと温存**できる（＝公開URL不変＝LINEもそのまま）。SSO必須の組織リポジトリではないことも確認。
3. **LINE**: プロバイダー `junsuke-personal` は**あなたの個人LINE**でログインする所有？（名前的に個人の想定）会社のLINE Business管理下でなければそのままでよい。

## 移行手順（推奨順）

### 心構え：データは失われにくい
各端末は localStorage に**完全なローカルコピー**を持っています。バックエンドを新Googleに切り替えても、**全データが揃っている端末が1台あれば**、新backendで「家庭を作成」→ もう片方が「参加」で復元できます。だから移行でデータが飛ぶ心配は小さい。念のため、切替前に**1台に最新が揃った状態**にしておくと万全。

### ① Google（必須の本丸）
1. **私物Googleアカウント**を用意（Gmail等。会社Workspaceでないもの）。
2. スプレッドシートを私物側へ。組織ポリシーでオーナー移管が不可なことが多いので、確実なのは **私物アカウントで新規スプレッドシートを作り、`households` シートを同名で用意**（会社側のデータをコピーしたい場合は data 列のJSONごとコピペ。前述のとおり端末から復元も可）。
3. 私物アカウントで**新規GASプロジェクト**を作成し、`backend/Code.gs` を貼付。
4. スクリプトプロパティを再設定：`SHEET_ID`（新シートのID）、`LINE_LOGIN_CHANNEL_ID`（**変わらず** `2010693415`）。
5. **ウェブアプリとして再デプロイ**（実行=自分／アクセス=全員）→ 新しい `/exec` URL を取得（※ `/u/N/` を含まない共有用URL）。
6. → 新URLを教えてもらえれば、私が `js/config.js` の `SYNC_URL` を更新してデプロイ。
7. 各端末で「家族と共有」を一度「解除」→ 新backendで作成/参加し直して復元。

### ② GitHub（多くはメール変更で完了）
- **まずSettings → Emailsで登録メールを私物に変更**し、会社メールを削除。2FA/復旧手段も私物に。これで**同じアカウント・同じ公開URLを温存**でき、LINEのendpoint更新も不要。
- どうしてもアカウントごと分けたい場合のみ：私物GitHubを作り、(a) リポジトリを Transfer ownership（履歴保持・会社アカウントにアクセスできるうちに）か、(b) 新規repoにpush。→ この場合は**公開URLが変わる**ので下の③のendpoint更新が必要。私が git remote と作者設定を付け替え。

### ③ LINE（②で公開URLを変えた場合のみ作業）
- **URLが変わらなければ何もしなくてよい。**
- 変えた場合：LINE Developers → LIFFの**エンドポイントURLを新しいPages URLに更新**（LIFF_ID・チャネルIDは同じでよい）。
- （プロバイダー自体が会社所有だった場合のみ）私物LINEでプロバイダー＋チャネルを作り直し → 新LIFF_ID/新チャネルID → `config.js` の `LIFF_ID` と GASの `LINE_LOGIN_CHANNEL_ID` を差し替え。影響大なので所有確認が肝。

### ④ 仕上げ
- 私が対応：`config.js`（SYNC_URL / 必要ならLIFF_ID）、`CLAUDE.md`・`docs`・`backend/README.md` のURL/アカウント記述の更新、git remote/author 付け替え、デプロイ。
- 動作確認：2台のスマホで作成→参加→相互反映まで。

## 役割分担

- **私**：コード内のURL/ID差し替え、ドキュメント更新、git設定、デプロイ。
- **あなた**：各アカウントの作成・移管・デプロイ・スクリプトプロパティ設定（認証情報が絡むので私は代行不可）。

## 将来の備え（任意）

公開URLを**独自ドメイン**（GitHub Pagesにカスタムドメイン割当。年1,000円程度）にしておくと、今後GitHubアカウントを変えても**アプリURL＝LIFFのendpointが不変**になり、以後の移行が楽。日常使いで長く使うなら検討価値あり。
