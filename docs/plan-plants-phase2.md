# 植物まわり本格強化の検討(2026-07-15・検討のみ実装なし)

依頼の3本柱:
1. **ChatGPT相談連携**(既存設計 `plan-ai-consult-history.md` の実装)
2. **月別お世話カレンダーDB**(植物・果物・野菜ごとに「何月に何をすべきか」を閲覧。**まだ家に無い植物も含む**図鑑的DB)
3. **植物の写真登録**(ChatGPTの画像が見られない場合の保険+成長記録)

これに加えて「植物のお世話」のユーザーニーズを拡散的に洗い出し、機能候補ごとに評価する。

---

## 1. ユーザーニーズの拡散的思考

家庭で植物を育てる人(=このアプリのユーザー像: 育児中の共働き夫婦)の立場で考えられるニーズ:

- **N1. 忘れない**: 水やり・植え替え・摘心などの「时期が来たら思い出したい」(既存の水やり周期+お手入れ予定+毎朝LINEで一部実現済み)
- **N2. 正しく知る**: 「この植物、いつ植え替えるのが正解?」という知識の参照。ネット検索は毎回面倒で結果もバラバラ
- **N3. 相談する**: 「葉が黄色いけど大丈夫?」という個別診断。汎用知識では答えが出ない(→ChatGPT連携)
- **N4. 記録する**: いつ何をしたかの履歴。「前回の植え替えいつだっけ」(careLogで一部実現済み)
- **N5. 成長を楽しむ**: 写真で成長を見返す、家族と共有して愛着を持つ。実用でなく情緒的価値
- **N6. 次を選ぶ**: 「次に何を育てよう」の検討材料。難易度・季節・置き場所との相性(→図鑑的DB)
- **N7. 買い物と連動**: 肥料・土・鉢が必要になったら買い物リストへ(既存の買い物リストと連動可能)
- **N8. 失敗から学ぶ**: 枯らしてしまった時に原因を振り返る(相談履歴+記録が揃えば副次的に実現)

## 3. 評価基準の設定

| 基準 | 内容 |
|---|---|
| K1. 価値の大きさ | ユーザー(家族)の生活が実際に良くなるか。使用頻度×効果 |
| K2. 実装コスト | フロント/バックエンド/外部設定の作業量 |
| K3. 技術リスク・制約 | localStorage容量、GASのセル上限(1セル5万文字)、外部API依存など |
| K4. 既存設計との整合 | 既存のstore構造・世帯同期・plantCareItems機構に自然に乗るか |
| K5. 保守・運用負担 | 作った後の手入れ(DB更新、トークン管理等)がどれだけ要るか |
| K6. 単独価値 | 他の機能が無くても、それ単独で価値が出るか(段階投入のしやすさ) |

## 4. 各機能の評価

### A. ChatGPT相談連携

- **K1 価値: ◎** 「葉が黄色い」等の個別相談は汎用DBでは代替不可。設計済みの`getPlantContext`(植物カルテ)が実現すれば、毎回状況説明が不要になり体験が突出する
- **K2 コスト: 高** GASにconsultationsシート+API(4本)、GPT Actions設定(ユーザー操作)、アプリに植物詳細画面+ルーターparam対応。フェーズ分割前提
- **K3 リスク: 中** GASはヘッダー認証不可→token-in-body方式で妥協(設計済み)。GPT Actions側の仕様変更リスクは外部依存として残る
- **K4 整合: ◎** 設計メモが既にあり、「相談履歴は独立コレクション」「書き込みはGPT側のみ」等の設計判断が済んでいる。世帯同期とも干渉しない(読み取り専用・別キャッシュ)
- **K5 保守: 中** GPT側の指示文・Actionsスキーマのメンテが必要
- **K6 単独価値: ◎** Phase1(GAS+GPT Actionsのみ)の時点で「スプレッドシートに相談が溜まる」価値が出る。アプリ改修ゼロで開始可能
- **総評**: 本命。ただし一気に作らずPhase分割(設計メモの実装フェーズ案どおり)

### B. 月別お世話カレンダーDB+図鑑UI

- **K1 価値: ◎** N2(正しく知る)は毎月発生する反復ニーズ。未所有植物も引ける図鑑はN6(次を選ぶ)にも効く
- **K2 コスト: 中** データは**リポジトリ内の静的JSON**(例: `js/plantpedia.js`)で持てばバックエンド不要。UIは図鑑一覧+詳細(月別カレンダー表示)の新画面2つ
- **K3 リスク: 低〜中** 技術リスクはほぼ無い。**リスクはデータの「質」**: AIで一括生成したお世話カレンダーの内容が不正確だと、かえって害になる(植え替え時期を間違える等)。私が下書きを生成し、渡辺さんが「うちにある植物だけでも」目視確認する運用が現実的
- **K4 整合: ○** 読み取り専用の静的データなので、store・世帯同期に一切影響しない。ただし後述Dの連動をやるなら、DBのスキーマ設計(作業種別のenum等)を既存careTasksのラベルと揃えておくべき
- **K5 保守: 中** 植物の追加要望が出るたびにJSONを足す作業が発生(ただしAI支援で軽い)
- **K6 単独価値: ◎** 閲覧だけでも十分役に立つ
- **総評**: コスパ最良。**最初に着手する候補**。データ品質担保のワークフローだけ決めてから始める

### C. 植物の写真登録

- **K1 価値: ○** 成長記録(N5)は情緒的価値が高く「妻に見せたい」系の機能。相談時の状態記録としても有用
- **K2 コスト: 中〜高** UI自体は`<input type="file" accept="image/*">`で撮影・選択でき簡単。**問題は保存先**(K3)
- **K3 リスク: 高(最重要論点)** 画像をbase64でlocalStorageに入れると: ①localStorage上限(約5MB)を写真数枚で食い尽くす ②世帯同期のdataに含めると**Googleスプレッドシートの1セル5万文字上限**を即超過し、同期が壊れる。→ **画像はlocalStorage/同期dataに入れず、GAS経由でGoogle Driveへアップロードし、URLだけを持つ**構成がほぼ必須。GASでのDriveアップロード+公開URL発行は実績のあるパターンだが、認可設定とサイズ制限(GASのPOST上限は約50MB、実用は数MB/枚に圧縮)の設計が要る
- **K4 整合: ○** `plant.photoUrls[]`(URL配列)を足すだけならstore影響は軽微。設計メモでも`photoUrl`は将来枠として想定済み
- **K5 保守: 低** 一度作れば手がかからない。Driveの容量はGoogleアカウントの無料枠(15GB)次第
- **K6 単独価値: ○** 写真だけでも成長記録として楽しめる
- **総評**: 価値はあるが、**保存先設計(Drive連携)が本体**。バックエンド作業になるのでAのGAS拡張と同じタイミングでやると効率的

### D. DBとお世話予定の連動(「今月は摘心の時期」自動表示)【実装済み・2026-07-15】

- **K1 価値: ◎** BのDBが「見に行けば分かる」なのに対し、Dは「向こうから教えてくれる」。N1×N2の掛け算で、このアプリの思想(開いた瞬間に分かる)に一番合う
- **K2 コスト: 低(B完了後)** 「うちの植物」の種類とDBを突き合わせ、当月に該当する作業を`plantCareItems`相当としてホーム・お知らせ・LINEダイジェストに合流させるだけ。既存機構に乗る
- **K3 リスク: 中** Bのデータ品質に完全依存。誤った時期を毎朝通知したら逆効果。最初は「控えめ表示(参考情報として)」から始めるのが安全
- **K4 整合: ◎** plantCareItems機構(導出データ・保存しない)にそのまま乗る設計が可能
- **K6 単独価値: ×** B無しでは成立しない(依存関係あり)
- **総評**: Bとセットで計画。B完成後の「2週目」として自然

**実装内容(v0.16.0)**:
- `App.data.plantPediaTips()`(store.js): うちの植物の名前と図鑑(`App.PLANTPEDIA`)をゆるく突き合わせ、今月が適期の作業を返す。すでに自分でお手入れ予定(careTasks)を登録済みのラベルは重複させない
- **お知らせ(ベル)**: 「植物」カテゴリの末尾に参考情報として合流(クリックで植物画面へ遷移。チェック機能は無し)
- **植物の記録画面**: 各植物カードに「今月のヒント」ボックスを追加表示(該当があるときのみ)
- **注意して据え置いた点**: 「今日やること」の完了チェックリスト(plantCareItems)には混ぜていない。図鑑の適期は月単位の目安でdone/undoneの概念がなく、チェックボックスに混在させると「押しても何も起きない」体験になるため、意図的に参考情報の場所(お知らせ・植物カードのみ)に留めた
- **LINE毎朝ダイジェストへの反映は未着手**: GASはフロントの`js/plantpedia.js`を直接参照できないため、反映するには図鑑データをGAS側にも複製する(保守負担増)か、公開JSをUrlFetchApp経由で毎回取得・パースする(壊れやすい)必要があり、今回は見送り。要望があれば改めて検討

### E. 図鑑から「うちの植物」への登録

- **K1 価値: ○** 新しい植物を迎えた時の登録が楽になり、種類情報が正規化される(Dの精度も上がる)
- **K2 コスト: 低** 図鑑詳細に「うちの植物に追加」ボタン→既存のopenPlantSheetに種類を事前入力するだけ
- **K4 整合: ◎** 既存の植物追加フローの入口が増えるだけ
- **K6 単独価値: △** B前提
- **総評**: Bのおまけとして同時実装が合理的

### F. 資材の買い物リスト連動

- **K1 価値: △** あれば便利だが、今でも買い物リストに手で「肥料」と書けば済む。自動化の伸びしろが小さい
- **K2 コスト: 低**
- **総評**: 優先度低。DBに「必要な資材」欄を持たせておき、詳細画面から1タップで買い物リストに送れる程度の軽い実装なら、B実装時のついでに検討

### G. 成長タイムライン(写真+careLog+相談履歴の統合ビュー)

- **K1 価値: ○** 全部が揃った時の「集大成」画面。振り返り(N8)と愛着(N5)に効く
- **K2 コスト: 中** A(相談履歴)とC(写真)が揃っていることが前提。植物詳細画面(Aで新設予定)の中の1セクションとして実装するのが自然
- **K6 単独価値: ×** A・C完了後にやる統合機能
- **総評**: 最後のお楽しみ。A・Cの設計時に「時系列で並べられるようデータに日時を必ず持たせる」ことだけ守っておく

## 5. 推奨の進め方(依存関係を踏まえた順序)

```
第1週: B(図鑑DB+閲覧UI) ← 単独で価値・技術リスク最小・最速で成果が見える
   +E(うちの植物に追加)を同時に
第2週: D(DBとお世話予定の連動) ← Bの上に低コストで乗る
第3週〜: A(ChatGPT連携)をPhase分割で
   Phase1: GAS+GPT Actions(アプリ無改修で相談蓄積が始まる)
   Phase2: 植物詳細画面+閲覧(ルーターparam対応が前提リファクタ)
同じ頃: C(写真登録) ← AのGAS拡張とまとめてDrive連携を作ると二度手間がない
最後: G(成長タイムライン) ← 全部の統合
```

- **Bを最初にする理由**: 依頼の3つの中で唯一バックエンド不要・外部設定不要で、渡辺さんの操作なしに私だけで完結できる。かつDの土台になる
- **AとCを同じ時期にする理由**: どちらもGASの拡張(consultationsシート/Driveアップロード)で、GAS側の作業をまとめられる
- **事前に決めておくべきこと**:
  1. Bのデータ品質担保: 初期収録する植物のリスト(うちにある植物+興味のある植物)と、内容の確認方法
  2. Cの写真は世帯で共有するか(Driveに置けば自然に共有可能だが、容量は渡辺さんのGoogleアカウントを消費)
  3. Aは既存設計メモの「リスク・決めごと」5点(plantId安定性・認証方式など)の最終確認
